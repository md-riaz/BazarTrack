 1. App Overview

1.1 Product Summary
সহজ বাজার খাতা is a collaborative office market management app. It replaces paper notebooks and
WhatsApp group chaos with a structured but flexible digital system for managing daily office and
personal shopping tasks, money balances, and expense records.


  Core Promise
  বাজারের তালিকা, খরচ, আর টাকা কার কাছে কত আছে — সব এক জায়গায় সহজভাবে।(Shopping lists,
  expenses, and who holds how much money — all in one simple place.)



1.2 Target Users & Roles
The app has exactly 3 roles — simple, clear, real-office-friendly.


 Role             Bangla            Who Uses This                 Core Responsibility

 admin            অ্যাডমিন          System administrator          Create wallets, assign owners, manage
                                                                  users, correct mistakes, monthly close

 owner            মালিক             CEO, CTO, Accounts            Manage assigned wallet(s) — balance,
                                    Officer, any designated       money entries, reports, edit bazar lists
                                    wallet manager

 assistant        সহকারী            Office assistant(s)           Shop for any wallet, enter prices, upload
                                                                  receipts, add/update items



  Real Office Setup Example
  Step 1 — Admin creates wallets: "Office", "CEO Personal", "CTO Personal"Step 2 — Admin
  assigns owners: CEO → "CEO Personal", Accounts Officer → "Office", CTO → "CTO
  Personal"Step 3 — All assistants automatically see every wallet and can shop for any of
  them.Result: CEO only sees his wallet. Accounts Officer only sees Office wallet. Assistants see
  all.



1.3 What This App Is NOT
   •​ Not an ERP or strict accounting software
   •​ Not a product inventory system
   •​ Not a barcode / POS system
   •​ Not a payroll system




 2. Core Concepts

2.1 Wallet (ওয়ালেট)

A Wallet replaces the older "Khata" concept. Each wallet represents a money pool for a specific
purpose or person.


 Wallet Name                         Type                   Typical Managers

 Office Wallet                       Shared                 Accounts Officer 1, Accounts Officer 2

 CEO Wallet                          Personal               CEO + assigned assistant

 CTO Wallet                          Personal               CTO + assigned assistant

 Office Lunch Wallet                 Shared                 Accounts team

 Event Wallet                        Temporary              Event coordinator


Key rules:
   •​ A wallet can have one or more managers (e.g. two Accounts Officers can both manage
      Office Wallet)
   •​ Any assistant can view and create expenses for a wallet they are permitted on
   •​ Wallet balance is NEVER stored — it is always dynamically calculated



2.2 Bazar (বাজার Session — Optional Grouping)
A Bazar is an optional grouping mechanism. Assistants are NOT required to create a bazar
before entering expenses.


  Two valid workflows
  (A) Quick Expense: Assistant goes to shop, comes back, adds expense directly to wallet.(B)
  Planned Bazar: Someone creates a list with items, assistant picks it up, buys items, closes it.


A Bazar can be:
   •​ Open (anyone available can take it)
   •​ Requested (CEO says "Rahim will do this bazar" — optional assignment)



2.3 Market Item
Each item inside a bazar requires only a name. Everything else is optional.


 Field                       Required?           Description

 name                        YES                 Item name in Bangla or English

 quantity                    No                  Amount to buy

 unit                        No                  packet, kg, piece, etc.

 attributes                  No                  Brand, type, notes e.g. "Aarong full cream"

2.4 Item Status Flow
 Status                    Bangla                        Meaning

 pending                   বাকি                          Not yet purchased

 done                      কেনা হয়েছে                   Price entered → auto-counted as expense

 not_found                 পাওয়া যায়নি                 Could not buy

 cancelled                 লাগবে না                      Removed from list


Note: No "partial" or "extra" status needed. Quantity can be updated freely. Activity log records all
changes.



2.5 Money Entry Types
 Type                             Bangla Label                 Balance Effect

 money_received                   টাকা নিলাম                   + (assistant received money from wallet)

 money_returned                   টাকা ফেরত দিলাম              - (assistant returned unused money)

 adjustment                       সমন্বয়                      +/- (admin correction, always needs note)


Note: Normal item purchase expense does NOT need a money_entry. It is auto-calculated from
item prices.




 3. Balance Calculation Logic

3.1 Core Formula



Confidential — For Development Use OnlyPage
সহজ বাজার খাতা — App Technical Specificationv1.0 | Confidential


  Balance (per wallet, per assistant) =
    SUM(money_received)
    - SUM(item prices where status = 'done')
    - SUM(direct_expenses)        ← expenses entered without a bazar
    - SUM(money_returned)
    +/- SUM(adjustments)

  IMPORTANT: Balance is NEVER stored in any table.
             It is ALWAYS calculated fresh from source entries.




3.2 Why Dynamic Calculation?
During shopping, entries can be wrong. Example: assistant enters "30 eggs = 200 BDT" but after
returning to office realizes it was "28 eggs = 180 BDT." Since balance is recalculated from entries,
editing the item automatically corrects the balance. No manual adjustment entry needed.



3.3 In-Progress vs Confirmed Balance
The app displays two balance figures when a bazar is still open:

 Balance Type                        Source                               Meaning

 Confirmed Balance                   Closed bazars + direct expenses +    Settled, reliable figure
                                     money entries

 In Progress                         Open/active bazars                   Tentative, may change

 Estimated Balance                   Confirmed + In Progress combined     Full picture



  UI Display Example
  CEO Wallet — RahimConfirmed Balance: +৳ 3,200In Progress:                 -৳ 850 (Rahim is
  currently shopping)Estimated Balance: +৳ 2,350



3.4 Balance Labels (Human Friendly)
 Condition                           Display Label

 Balance > 0                         Rahim-এর হাতে আছে ৳ X (Assistant has money remaining)

 Balance < 0                         Rahim পাবে ৳ X (Assistant is owed money)

 Balance = 0                         হিসাব মিলে গেছে (Settled)




 4. Edit Rules & Audit Safety

4.1 Three-Layer Edit Control




Confidential — For Development Use OnlyPage
সহজ বাজার খাতা — App Technical Specificationv1.0 | Confidential


 Layer                     Trigger                                Effect

 Edit Window               Bazar still open (চলতেছে)              All items freely editable

 Soft Lock                 Bazar closed (বন্ধ)                    Only Admin can edit; every edit is
                                                                  logged

 Hard Lock                 Monthly period closed by Accounts      Nobody can edit; permanent


Money entries (money_received, money_returned) can be edited only on the same day they were
created. After that, a correction requires an adjustment entry with mandatory note.



4.2 Monthly Closing (হিসাব বন্ধ)
At end of each month, the Accounts Officer performs a monthly close:
   1.​ Accounts Officer reviews all entries for the month
   2.​ Confirms balance with assistant / wallet owner
   3.​ Clicks "হিসাব বন্ধ করুন" in the app
   4.​ System creates a wallet_snapshot record with closing balance
   5.​ All entries for that period become permanently locked (is_locked = true)


  wallet_snapshots table
  ─────────────────────────────────────
  id                UUID
  wallet_id         FK
  assistant_id      FK (nullable — per-assistant snapshot)
  period_month      YYYY-MM
  opening_balance   DECIMAL
  closing_balance   DECIMAL
  snapshot_hash     VARCHAR(64)   ← SHA-256 tamper detection
  closed_by         FK users
  closed_at         TIMESTAMP
  notes             TEXT




4.3 Balance Calculation After Monthly Close
  Current Balance =
    Last Closed Snapshot.closing_balance
    + SUM(entries since snapshot.closed_at)

  This limits recalculation to current month only → fast query.




4.4 Tamper Detection
On every balance page load, the system re-hashes the locked period entries and compares with
stored hash. If mismatch is detected:
   •​ Admin receives an alert
   •​ Affected balance display shows a warning badge
   •​ System logs the discrepancy in audit_alerts table

 6. Roles & Permissions

6.1 Role Summary


  The 3-Role Mental Model
  admin → God mode. Creates structure. No shopping.owner → Manages their wallet(s). Sees
  only what is assigned to them.assistant → Does the shopping. Sees all wallets unless a bazar is
  specifically assigned to someone else.



6.2 Wallet Visibility Rules


 Who                             Which wallets can they see?

 admin                           ALL wallets

 owner                           ONLY wallets they are assigned to (via wallet_members table)

 assistant                       ALL wallets by default



6.3 Bazar Visibility Rules (Important)


 Bazar assigned_to field                       Who can see the bazar?

 NULL (open, no assignment)                    All assistants + wallet owners + admin

 Specific assistant (e.g. Rahim)               Only Rahim + wallet owners + admin. Other assistants
                                               CANNOT see it.



  Why this rule?
  If CTO specifically says "Rahim will do my shopping today", other assistants should not interfere
  or see the private items in that bazar. The assignment creates a privacy boundary.

 12b. Offline-First Architecture

Offline-First is a v1 Requirement — Not v2
Office assistants often shop in areas with poor connectivity (markets, basement shops). The app
must work fully offline and sync when internet returns. This is not a luxury feature — it is essential
for real-world use.



Offline Architecture Overview
  WRITE PATH (always goes to local DB first):
    User action → Drift local DB → UI updates immediately (optimistic)
                                → Sync queue entry added
                                → Background sync when online → Server DB

  READ PATH:
    App reads from Drift local DB always
    Server data syncs down in background

  SYNC ENGINE:
    SyncQueue table tracks all pending writes
    Retries on connectivity restore
    Conflict resolution: last-write-wins by updated_at timestamp
    Server is source of truth for conflict resolution




Local Database Schema (Drift — SQLite)
  sync_queue                          ← tracks pending writes
  ──────────────────────────────────────────────
  id            INTEGER    PRIMARY KEY AUTOINCREMENT
  entity_type   TEXT       bazar_item | money_entry | bazar | direct_expense
  entity_id     TEXT       UUID of the record
  operation     TEXT       create | update | delete
  payload       TEXT       JSON of the full record
  created_at    INTEGER    Unix timestamp
  retry_count   INTEGER    DEFAULT 0
  last_error    TEXT       NULLABLE

 14. Final Notes for Developers


  Critical Design Principles — Do Not Compromise
  1. Balance is NEVER stored. Always calculated dynamically.2. Bazar is OPTIONAL grouping.
  Quick expense must always work without it.3. Assignment is OPTIONAL. Wallet stays open for
  any member unless specifically assigned.4. Same-day edit only for money entries. After that,
  adjustment entry required.5. Monthly close = hard lock (is_locked). SHA-256 hash verification is
  planned v1.5 — do NOT build it in MVP.6. All schema changes must be backward compatible.
  Never delete columns, always soft delete.7. New Bazar must ALWAYS offer both text-parse
  mode AND manual one-by-one mode. Neither is optional.8. Offline-first is non-negotiable from
  v1. Do not defer it. Do not build online-only first.



  Questions to Clarify Before Starting Development
  1. How many wallets are expected? (affects permission design)2. Will both Android and iOS be
  needed, or Android only for MVP?3. What server/hosting is available? (shared hosting won't
  support Redis/Queue)4. Is receipt photo required before closing a bazar or optional always?5.
  Monthly close — who initiates: wallet owner or admin only?6. Should assistants see other
  assistants' balances or only their own?
